# WORKFLOW.md — GitHub 작업 가이드

> 비개발자를 위한 GitHub 웹 UI 전용 가이드입니다. Git 명령어 없이 브라우저만으로 작업합니다.

---

## 브랜치 구조

```
main                          ← 배포 브랜치 (항상 동작하는 상태 유지)
└─ dev                        ← 개발 통합 브랜치
   ├─ feat/기능명              ← 새 기능 개발
   ├─ fix/버그명               ← 버그 수정
   └─ docs/문서명              ← 문서 작업
```

**규칙**
- `main`에 직접 push하지 않는다
- 모든 작업은 브랜치에서 시작해 PR로 합친다
- 기능이 완성되면 `feat/*` → `dev` → `main` 순서로 합친다

---

## 브랜치 이름 짓기

| 작업 종류 | 형식 | 예시 |
|-----------|------|------|
| 새 기능 | `feat/기능명` | `feat/login-page` |
| 버그 수정 | `fix/버그명` | `fix/upload-error` |
| 문서 작업 | `docs/문서명` | `docs/readme-update` |
| UI 개선 | `style/작업명` | `style/interview-screen` |

---

## 새 브랜치 만들기 (GitHub 웹 UI)

1. 레포 메인 페이지에서 브랜치 드롭다운 클릭 (`main` 또는 `dev` 표시된 버튼)
2. 검색창에 새 브랜치 이름 입력 (예: `feat/login-page`)
3. "Create branch: feat/login-page **from** dev" 클릭
4. 완료 — 이제 이 브랜치에서 작업 시작

---

## 파일 수정하기

1. 작업할 브랜치로 이동 (브랜치 드롭다운에서 선택)
2. 수정할 파일 클릭
3. 우상단 연필 아이콘(✏️) 클릭
4. 내용 수정
5. 하단 "Commit changes" 클릭

---

## 커밋 메시지 형식

```
타입: 한 줄 요약

(선택) 부연 설명이 필요하면 한 줄 띄고 작성
```

| 타입 | 언제 쓰나 | 예시 |
|------|-----------|------|
| `feat` | 새 기능 추가 | `feat: 로그인 화면 추가` |
| `fix` | 버그 수정 | `fix: 파일 업로드 오류 수정` |
| `docs` | 문서 수정 | `docs: README 업데이트` |
| `style` | UI/스타일 수정 | `style: 버튼 색상 변경` |
| `refactor` | 기능 변경 없이 코드 정리 | `refactor: 인터뷰 컴포넌트 구조 정리` |
| `chore` | 설정, 패키지 등 | `chore: 환경변수 추가` |

## PR (Pull Request) 만들기

작업한 브랜치를 `dev`에 합치고 싶을 때

1. 레포 상단 "Pull requests" 탭 클릭
2. "New pull request" 클릭
3. **base**: `dev` / **compare**: 내가 작업한 브랜치 선택
4. 제목과 설명 작성

**PR 제목 형식**
```
feat: 로그인 화면 구현
fix: 파일 업로드 오류 수정
```

**PR 설명 템플릿**
```
## 작업 내용
- 구현하거나 수정한 내용을 항목으로 작성

## 확인 사항
- [ ] 정상 동작 확인
- [ ] 에러 케이스 확인

## 참고
관련 이슈 번호, 링크 등
```

5. "Create pull request" 클릭
6. 확인 후 "Merge pull request" 클릭 → "Confirm merge"

## Antigravity 에이전트 작업 후 처리

에이전트가 코드를 생성한 뒤 해야 할 일:

1. 에이전트가 생성한 파일 변경 내역 확인
2. 동작 확인 (Vercel 프리뷰 링크 또는 로컬)
3. 문제 없으면 커밋 메시지 작성 후 커밋
4. PR 생성 → `dev`로 머지

---

## 자주 쓰는 용어 정리

| 용어 | 뜻 |
|------|----|
| **레포 (Repository)** | 프로젝트 코드가 저장된 공간 |
| **브랜치 (Branch)** | 독립적으로 작업하는 코드 사본 |
| **커밋 (Commit)** | 변경 내용을 저장하는 행위. "저장 + 기록" |
| **PR (Pull Request)** | 내 브랜치를 다른 브랜치에 합쳐달라는 요청 |
| **머지 (Merge)** | 두 브랜치를 합치는 것 |
| **main** | 실제 서비스에 배포되는 최종 브랜치 |
| **dev** | 개발 중인 코드를 모아두는 브랜치 |
| **feat/** | 새 기능을 만드는 브랜치 |

## 비상 상황 대처

**잘못된 브랜치에서 작업했을 때**
→ 당황하지 말고 그대로 커밋 후 올바른 브랜치로 PR 생성

**파일을 실수로 삭제했을 때**
→ GitHub History에서 이전 버전 복원 가능. 파일 클릭 → "History" 탭 확인

**모르겠을 때**
→ `main` 브랜치는 건드리지 않는다. `dev`에서 새 브랜치 만들어서 재시작
