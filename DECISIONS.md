## 1. 클라이언트 사이드 문서 파싱

**결정**: PDF/DOCX 파싱을 서버가 아닌 브라우저(클라이언트)에서 처리한다
- 라이브러리: `pdf.js` (PDF), `mammoth.js` (DOCX)
- 추출된 텍스트만 Supabase DB에 저장, 원본 파일은 Storage에 보관

**이유**
- Vercel 무료 티어의 함수 실행 시간(10초) + 메모리 제한 대응
- 파일 크기가 클 경우 서버에서 파싱하면 타임아웃 위험
- 브라우저 파싱은 서버 비용 없이 처리 가능

**트레이드오프**
- 스캔본 PDF, 이미지 기반 문서는 파싱 불가 → 텍스트 직접 붙여넣기 옵션 병행 제공
- 클라이언트 성능에 따라 파싱 속도 차이 발생 가능

## 2. Next.js API Route를 Gemini 프록시로 사용

**결정**: Gemini API를 클라이언트에서 직접 호출하지 않고, Next.js API Route를 경유한다

**이유**
- Gemini API 키가 클라이언트 코드에 노출되면 탈취 위험
- 서버 환경변수에만 키를 저장하고 클라이언트는 내부 API만 호출
- 추후 요청 로깅, rate limiting, 인증 검증을 한 곳에서 처리 가능

**구조**
```
클라이언트 → /api/interview (Next.js API Route) → Gemini API
```

## 3. 폴백 없는 에러 처리 정책

**결정**: 에이전트 호출 실패 시 3회 재시도 후 하드 실패. 폴백(일부 기능 제외 진행) 없음

**이유**
> 불완전한 면접 경험은 실전 대응에 치명적이다.

- 절반만 준비된 질문으로 면접을 진행하면 오히려 잘못된 연습
- 에러가 발생했을 때 유저에게 명확하게 알리는 것이 더 나은 UX
- 재시도 로직으로 일시적 장애는 커버 가능

**구현**
- 각 에이전트: 최대 3회 재시도 (지수 백오프 적용)
- 3회 후 실패: 에러 메시지 + 재시작 버튼 표시

## 4. Google ADK SequentialAgent 오케스트레이션

**결정**: 3개 에이전트를 Google ADK의 SequentialAgent로 오케스트레이션한다

**에이전트 구조**
```
SequentialAgent (오케스트레이터)
├─ LlmAgent: 분석 에이전트
├─ LlmAgent: 면접관 에이전트
└─ LlmAgent: 평가 에이전트
```

**이유**
- 분석 → 면접 → 평가의 순서 의존성이 명확함
- ADK가 Gemini 3 Pro 네이티브 지원 + TypeScript 지원으로 Next.js 스택과 자연스럽게 통합
- 내장 Session Management로 면접 대화 이력 유지 용이

**실행 패턴**
- 분석 에이전트: 면접 시작 전 백그라운드 비동기 실행
- 면접관 에이전트: 면접 진행 중 동기 실행 (스트리밍)
- 평가 에이전트: 면접 종료 후 비동기 실행

## 5. 스트리밍 응답 처리

**결정**: Gemini API 응답을 스트리밍으로 처리한다

**이유**
- Vercel 무료 티어 함수 타임아웃(10초) 대응: 스트리밍은 첫 청크만 10초 내에 오면 됨
- 면접관이 실제로 말하는 듯한 타이핑 UX 구현
- 전체 응답 대기 없이 즉각적인 피드백으로 몰입감 유지

**구현**
```
Gemini API → SSE (Server-Sent Events) → 클라이언트 스트리밍 렌더링
```

## 6. 세션 상태: 메모리 + DB 혼합 저장

**결정**: 면접 진행 중 상태는 브라우저 메모리, 중요 시점에만 DB에 저장한다

**DB 저장 시점**
- 면접 시작 시 (시작 시각, 설정값)
- 매 답변 완료 시 (대화 이력)
- 브라우저 이탈 감지 시 (visibilitychange 이벤트)
- 면접 종료 시 (전체 이력 + 리포트)

**이유**
- 매 상태 변경마다 DB 저장하면 불필요한 API 호출 과다
- 브라우저 메모리가 가장 빠르고 Supabase 무료 티어 API 제한에 여유 확보
- 이탈 감지로 데이터 유실 없이 이어하기 지원 가능

## 7. 꼬리질문 판단: AI 전적 판단으로 변경

**결정**: 초기 설계의 "룰 기반 1차 필터 + AI 2차 판단" 구조를 "AI 전적 판단"으로 단순화

**초기 설계** (TRD v1)
- 1차: 수치/사례 없음, 답변 길이 미달, 심층 키워드 감지 → 룰 기반 플래그
- 2차: 1차 결과를 컨텍스트로 AI 최종 판단

**변경된 설계** (Prompt Design v1)
- 룰 기반 필터 제거, AI가 직접 꼬리질문 필요 여부 판단

**이유**
- 룰 기반 필터는 오탐(false positive) 가능성 높음 (예: 짧지만 훌륭한 답변에도 플래그)
- Gemini 3 Pro의 판단력이 충분히 신뢰할 수 있는 수준
- 코드 복잡도 감소, 유지보수 용이

**꼬리질문 판단 기준 (AI에 전달)**
- 답변이 모호하거나 추상적인가?
- 수치나 구체적 사례가 없는가?
- 더 파고들 만한 흥미로운 키워드가 있는가?
- 답변이 불충분한가?
- 현재 depth가 4 미만인가?

## 8. 질문 수: 150% 버퍼 생성

**결정**: 면접 시간 대비 150% 분량의 질문을 사전 생성한다

**계산식**: `duration / 5 * 1.5`
- 예: 60분 면접 → 60/5 = 12개 기준 → 18개 생성

**이유**
- 꼬리질문이 많이 발생하면 본 질문 소화량이 줄어듦
- 시간이 남았을 때 질문 부족 상황 방지
- 남은 시간 20% 이하 시 마무리 질문으로 자동 전환하는 로직과 연계

## 9. MVP 동일 가중치 평가

**결정**: MVP에서는 모든 질문 유형(공통/프로젝트/우대사항 갭)에 동일 가중치 적용

**이유**
- 가중치 설계는 실제 데이터 없이 주관적 설정이 될 위험
- 베타 테스트 후 실제 사용 패턴 확인 후 v2에서 조정
- MVP 단순화로 개발 속도 확보

**v2 방향**
- 프로젝트 질문 > 공통 질문 가중치 부여 (직무 인터뷰 특성 반영)
- 질문 depth에 따른 가중치 차등 검토

## 10. Git 파싱: README만 (코드 분석 제외)

**결정**: Git 링크 입력 시 README.md + 언어 비율 + 주요 스택만 파싱. 실제 코드/커밋 분석은 v2

**이유**
- 코드 분석은 GitHub API rate limit + 토큰 소비량이 큼
- README만으로도 프로젝트 컨텍스트를 충분히 확보 가능
- MVP 범위 내에서 기술 부채 없이 구현 가능

**v2 방향**
- 커밋 패턴 분석 (기여도, 작업 방식)
- 주요 파일 구조 분석
- PR/이슈 내용 분석
