## 1. 배경 및 문제 정의

### 1.1 문제

취준생은 면접을 준비할 때 모의면접을 진행할 수 있는 환경 자체가 없다. 사람과 함께하는 면접 스터디는 일정 조율이 어렵고 피드백 품질도 일정하지 않다. 유료 컨설팅은 비용 부담이 크다.

> **핵심 문제: 사람 의존 없이, 언제 어디서든 모의면접을 할 수 있는 환경이 없다.**

### 1.2 솔루션

실제 면접은 **직무(역량) 인터뷰 + 컬쳐핏 인터뷰** 두 트랙으로 나뉜다. 직무 인터뷰는 내 이력서/포트폴리오를 읽은 면접관이 진행한다. 기존 서비스들은 "나를 모르는 면접관"과의 generic한 연습만 제공한다.

리허설은 JD + 이력서 + 포트폴리오를 학습한 **"나를 아는 AI 스페셜리스트 면접관"** 과의 모의면접 환경을 제공한다.

### 1.3 서비스명 의미

> **리허설 (reHEARsal)**
>
> - 발음을 흘리면 **"이력서"** 가 들린다
> - RE + **HEAR** = "다시 듣게 하다" → 면접관이 듣고 싶은 말을 다시 들려주는 연습
> - 한자로 **履說(리설)** = 이력을 말하다

## 2. 목표

### 2.1 제품 목표

- IT 직군 경력직/신입 취준생이 언제 어디서든 실전과 유사한 모의면접을 경험할 수 있게 한다
- JD와 내 이력에 기반한 맥락 있는 질문으로 실질적인 연습 효과를 만든다
- 면접 후 구체적인 피드백으로 개선 방향을 제시한다

### 2.2 MVP 핵심 지표 (KPI)

MVP 기간을 **론칭 후 4주**로 정의하며, 아래 수치를 성공 기준으로 삼는다.

|지표|측정 방법|목표 수치|경고 수치|
|---|---|---|---|
|**면접 완주율**|면접 시작 이벤트 대비 종료 이벤트 발생 비율|≥ 65%|< 45%|
|**피드백 열람률**|리포트 화면 진입 후 30초 이상 체류한 비율|≥ 75%|< 50%|
|**7일 내 재방문율**|첫 면접 완료 후 7일 이내 면접을 1회 이상 재진행한 비율|≥ 30%|< 15%|

**경고 수치 도달 시 액션:**

- 완주율 < 45% → 면접 진행 중 이탈 지점 분석, 면접 시간 설정 기본값 검토
- 피드백 열람률 < 50% → 리포트 화면 첫 인상 UX 개선 우선 검토
- 재방문율 < 15% → 첫 면접 리포트 품질 집중 개선 (평가 프롬프트 튜닝)

## 3. 타겟 유저

### 3.1 Primary

**IT 직군 경력직 이직 준비자**

- 직군: 개발자, PM, 디자이너, AI 엔지니어, 인프라 엔지니어
- 서류 합격 후 면접을 앞둔 시점
- 실전 면접 준비를 위한 맞춤형 모의면접이 필요한 사람

### 3.2 Secondary

**IT 직군 신입 취준생**

- 면접 경험이 부족하고 피드백 받을 곳이 없는 사람
- 포트폴리오가 없거나 Git 링크만 있는 경우도 진입 가능해야 함

### 3.3 트리거 시점

서류 합격 직후 ~ 면접 전날

## 4. 범위 (Scope)

### 4.1 MVP In Scope

**[인증]**

- Google 소셜 로그인

**[온보딩]**

- 최초 로그인 시 환영 화면 + 서비스 핵심 가치 1문장 소개
- 이력서 업로드 유도 (빈 상태에서 면접 시작 불가 → 업로드로 자연 유도)

**[문서 관리]**

- 이력서/경력기술서, 포트폴리오 업로드 후 계정에 저장
- 언제든 삭제/재업로드 가능 (삭제 시 DB + Storage 완전 삭제)

**[입력]**

- JD: 텍스트 직접 입력 또는 URL 크롤링
- 이력서/경력기술서: 필수 (PDF, DOCX)
- 포트폴리오: 선택 ("포트폴리오 없음" 옵션 제공)
- Git 링크: 선택 (URL만 저장)

**[면접 설정]**

- 면접 시간: 30분 / 60분 / 90분 선택 (기본값 없음, 반드시 선택)
- 면접관 페르소나: 경험 탐색형 / 심층 압박형 (2종 선택)
- 면접 제목: 직접 입력 (히스토리 구분용)

**[시뮬레이션]**

- AI 면접관과 텍스트 기반 대화형 직무 인터뷰
- AI 면접관 음성 출력 (TTS, 기본값 켜짐 / 토글 가능)
- 유저 마이크 음성 입력 (STT → 텍스트 변환 후 답변으로 전송)
- 유저 답변에 따른 꼬리질문 (explorer 최대 2단계, pressure 최대 4단계)
- 힌트 버튼 온디맨드 (이력서 기반 AI 실시간 모범 답안 생성)
- 중간 이탈 시 이어하기 가능

**[환경설정]**

- 마이크 / 스피커 장치 선택
- 마이크 음량 / 스피커 음량 조절
- 마이크 테스트 (실시간 레벨 미터)

**[AI 모델 설정]** ← v3.0 신규

- 기본 제공: gemini-2.5-flash (서버 키, 별도 설정 불필요)
- 사용자 Gemini API 키 등록 시 상위 모델 선택 가능
- Settings 페이지에서 API 키 등록/삭제

**[리포트]**

- 답변별 평가: 논리성 / 구체성 / 직무 적합성 (각 100점)
- 전체 총점 및 총평
- 질문별 모범 답안 (본 질문 + 꼬리질문 각각)
- 재도전 추천 질문

**[히스토리]**

- 면접 기록 최근 10회 저장
- MVP 전체 무료

### 4.2 MVP Out of Scope (v2 이후)

|항목|이유|
|---|---|
|컬쳐핏 인터뷰|직무 인터뷰 품질 검증 먼저|
|영상 녹화 및 표정 분석|인프라 비용, 개인정보 처리 복잡도|
|Git 코드/커밋 분석|MVP는 URL 저장으로 충분|
|모바일 앱|웹 먼저, 사용 패턴 확인 후 결정|
|BM / 유료 플랜|사용자 확보 후 전환율 데이터 기반으로 설계|
|멀티 프로바이더 (OpenAI·Claude)|Gemini BYOK로 먼저 시작, 수요 확인 후 확장|

## 5. 유저 플로우

### 5.1 첫 방문자 플로우 (신규 가입 ~ 첫 면접)

```
[STEP 1] 랜딩 및 로그인
  └─ Google 소셜 로그인
  └─ 최초 로그인 시 → 온보딩 화면 진입
     └─ 서비스 핵심 가치 1문장
     └─ "시작하기" 버튼 → STEP 2로 이동

[STEP 2] 문서 업로드 (최초 1회 필수)
  └─ 이력서/경력기술서 업로드
  └─ 포트폴리오 업로드 (선택)
  └─ Git 링크 입력 (선택)
  └─ 업로드 완료 → STEP 3으로 이동

[STEP 3] 면접 설정
  └─ 면접 제목 입력
  └─ JD 입력
  └─ 면접 시간 선택 (30분 / 60분 / 90분)
  └─ 면접관 페르소나 선택 (2종)
  └─ "면접 시작하기" 버튼 → STEP 4로 이동

[STEP 4] AI 컨텍스트 분석 (약 10~20초)
  └─ 분석 중 로딩 화면 표시
  └─ 분석 완료 → STEP 5로 자동 전환

[STEP 5] 모의면접 시뮬레이션
  └─ AI 면접관 인트로 멘트 (TTS 음성 출력)
  └─ 질문 → 유저 답변 → 꼬리질문 반복
  └─ 힌트 버튼 온디맨드
  └─ 중간 이탈 감지 → 이어하기 저장
  └─ 시간 종료 또는 유저 직접 종료

[STEP 6] 리포트
  └─ 평가 중 로딩 표시 (약 10~15초)
  └─ 총점 및 총평, 강점/개선점
  └─ 답변별 평가 + 모범 답안
  └─ 재도전 추천 질문
```

### 5.2 재방문자 플로우 (문서 보유 상태)

```
[로그인]
  └─ 홈 화면: 최근 면접 기록 + "새 면접 시작하기" 버튼
  └─ 미완료 세션 있을 경우 → "지난 면접 이어하기" 배너 노출

[새 면접]
  └─ STEP 3 (면접 설정)부터 시작

[이어하기]
  └─ "이어서 진행하시겠습니까?" 확인
  └─ 확인 시 → 마지막 질문 위치에서 재개
  └─ 거절 시 → 홈 화면으로 이동 (세션 abandoned 처리)
```

### 5.3 빈 상태 (Empty State) 처리

|상황|표시 내용|
|---|---|
|로그인 직후, 이력서 없음|"아직 이력서가 없어요. 이력서를 올리면 면접을 시작할 수 있어요" + 업로드 버튼|
|히스토리 없음|"아직 면접 기록이 없어요. 첫 면접을 시작해볼까요?" + 면접 시작 버튼|
|리포트 생성 중|스피너 + 진행 메시지 ("면접 내용을 분석하고 있어요")|

## 6. 기능 명세

### 6.1 문서 관리

|항목|상세|
|---|---|
|지원 형식|PDF, DOCX|
|파일 크기 제한|파일당 10MB 이하|
|저장 방식|계정에 저장, 면접 시 재사용|
|파싱 위치|**서버 사이드** (pdf-parse, mammoth — Server Actions)|
|삭제|DB + Storage 동시 완전 삭제|
|Git 파싱 범위|URL만 저장 (코드/커밋 분석은 v2)|

**엣지 케이스:**

|상황|처리 방식|
|---|---|
|파싱 실패 (손상 파일, 이미지 기반 PDF)|에러 메시지 + 텍스트 직접 붙여넣기 옵션 제공|
|10MB 초과 파일 업로드 시도|업로드 전 클라이언트에서 차단, 용량 초과 안내|
|동일 타입 문서 재업로드|기존 파일 교체 여부 확인 후 처리|
|스캔본 PDF / 이미지 이력서|파싱 불가 안내 + 텍스트 직접 붙여넣기 옵션으로 안내|
|Git URL 접근 불가|오류 안내 + "Git 링크 없이 진행하기" 옵션 제공|

### 6.2 JD 입력

|항목|상세|
|---|---|
|입력 방식|텍스트 직접 입력 또는 URL 크롤링|
|최소 글자 수|50자 이상|
|URL 크롤링 지원|주요 채용 플랫폼 (원티드, 로켓펀치, 사람인, LinkedIn 등)|

**엣지 케이스:**

|상황|처리 방식|
|---|---|
|URL 크롤링 실패|실패 안내 + 텍스트 직접 입력으로 자동 전환|
|JD 텍스트 너무 짧음 (50자 미만)|저장 전 경고 메시지|
|JD 없이 면접 시작 시도|JD 없는 면접으로 진행 (지원동기 질문 생략)|

### 6.3 면접 설정

|항목|상세|
|---|---|
|면접 제목|직접 입력 필수 (히스토리에서 구분용)|
|면접 시간|30분 / 60분 / 90분 중 선택 (기본값 없음, 선택 필수)|
|페르소나|2종 (경험 탐색형 / 심층 압박형)|
|페르소나 커스텀 지침|각 페르소나에 유저가 추가 지침 작성 가능|
|시작 조건|제목 + 이력서 선택 + 페르소나 + 시간 모두 선택해야 시작 버튼 활성화|

### 6.4 면접관 페르소나

|페르소나 (코드값)|이름|분위기|꼬리질문 상한|
|---|---|---|---|
|`explorer`|경험 탐색형|편안하고 대화적|최대 2단계|
|`pressure`|심층 압박형|날카롭고 도전적|최대 4단계|

**커스텀 지침:**

- 유저가 각 페르소나에 추가 지침을 직접 작성할 수 있음
- 작성 위치: 사이드바 > 면접관 페르소나 메뉴
- AI 면접관은 기본 지침 + 유저 커스텀 지침을 합산해 면접 진행
- 커스텀 지침 미작성 시 기본 지침만 적용

### 6.5 힌트 기능

- 노출 방식: "힌트 보기" 버튼 클릭 시 AI가 실시간으로 이력서 기반 모범 답안 생성
- 힌트 내용: 이력서에 실제로 기재된 내용만 근거로 한 모범 답안 (STAR 기법 활용)

**엣지 케이스:**

|상황|처리 방식|
|---|---|
|힌트 사용이 점수에 미치는 영향|**있음** — 힌트를 사용한 질문의 최종 점수는 최대 40점으로 제한됨. 리포트에 "모범 답안을 참조한 답변입니다" 명시|
|힌트 사용 횟수 제한|없음. 횟수 제한 없이 자유롭게 사용 가능|
|꼬리질문에서의 힌트|꼬리질문에도 동일하게 힌트 제공|

### 6.6 음성 기능 (TTS / STT)

**TTS — AI 면접관 음성 출력**

|항목|상세|
|---|---|
|모델|Gemini 2.5 Flash Preview TTS (항상 서버 키 사용)|
|기본값|켜짐 (면접 시작 시 자동 활성화)|
|토글|면접 진행 중 음소거 버튼으로 언제든 켜고 끄기 가능|
|재생 속도|1.3× (코드 고정값)|
|오류 처리|TTS 실패 시 텍스트만 표시하고 면접 계속 진행 (non-critical)|

**STT — 유저 마이크 음성 입력**

|항목|상세|
|---|---|
|모델|Gemini 2.5 Flash 전사 (항상 서버 키 사용)|
|입력 형식|WebM (audio/webm;codecs=opus)|
|처리 방식|녹음 완료 후 `/api/transcribe`로 전송 → 전사 텍스트를 답변창에 자동 입력|
|텍스트 입력 병행|항상 가능. 마이크 없이 키보드로만 사용해도 무방|

**엣지 케이스:**

### 6.8 이어하기

- 이탈 감지: `handleExit()` 명시적 호출 시 남은 시간 저장
- 이어하기 유효 기간: **이탈 후 24시간 이내**
- 24시간 경과 시: 세션 `abandoned` 처리, 이어하기 불가

**엣지 케이스:**

|상황|처리 방식|
|---|---|
|재접속 시 미완료 세션 있음|홈 화면 상단에 "지난 면접 이어하기" 배너 노출|
|이어하기 거절 시|세션 abandoned 처리, 리포트 생성 안 함|
|분석 단계(STEP 4)에서 이탈|분석 완료 전이므로 이어하기 불가, 면접 설정부터 재시작 안내|

**특수 처리:**

**엣지 케이스:**

### 6.11 AI 모델 설정 — v3.0 신규

사용자가 자신의 Gemini API 키를 등록하면 기본 모델보다 성능이 높은 상위 모델로 면접을 진행할 수 있다.

**지원 모델:**

|모델|제공 방식|특징|
|---|---|---|
|gemini-2.5-flash|서버 기본 키 (무료)|기본값. 별도 설정 불필요|
|gemini-2.5-pro|사용자 API 키 필요|향상된 분석·평가 품질|
|gemini-3.1-pro-preview|사용자 API 키 필요|최고 성능, 심층 꼬리질문 품질 향상|

**Settings 페이지 구성:**

- 현재 상태 표시: "기본 모델 사용 중" / "내 API 키 사용 중 — (모델명)"
- Gemini API 키 입력창 (저장 시 AES-256 암호화)
- 모델 선택 드롭다운
- 저장 / 키 삭제 버튼
- 안내 문구: "API 키는 AES-256 암호화 후 저장되며 서버에서만 복호화됩니다. 음성 기능(TTS/STT)은 항상 기본 키를 사용합니다."

**엣지 케이스:**

|상황|처리 방식|
|---|---|
|유효하지 않은 API 키 입력|저장 전 유효성 검사 → "유효하지 않은 API 키" 에러 (저장 안 됨)|
|API 키 삭제|기본 모델(gemini-2.5-flash)로 자동 복귀|
|키 등록 상태에서 면접 시작|등록된 키 + 선택 모델로 면접 진행|
|TTS/STT는 키 설정과 무관|항상 서버 기본 키 사용 (gemini-2.5-flash 계열)|
